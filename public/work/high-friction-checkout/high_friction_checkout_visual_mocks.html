Checkout migration case study

A banner that was secretly a state machine.

I migrated a loyalty enrollment flow from legacy PHP/React checkout into a schema-driven Block Builder experience, making it configurable without making checkout fragile.

Before

One blob of checkout-specific data

PHP decided the copy, images, eligibility, rewards values, and UI states. React mostly rendered a custom one-off shape.

{ "potentialRewards": "$18.42", "isLoyaltySkuAdded": false, "isInSuppressionPeriod": false, "logoImageId": "...", "default": { "title": "Earn rewards today", "choice1": "Yes, enroll me", "choice2": "No thanks", "cta": "Join Rewards" }, "added": { "...": "..." }, "declined": { "...": "..." } }
Problem Changing copy/layout required code because content, business data, UI structure, and behavior were tangled together.
After

Schema for content, state for behavior

Block Builder owns the shape of the experience. Checkout state owns live cart data. The reducer owns interaction flow.

{ "__typename": "BlockBuilderLoyaltyProgramHighFrictionCheckoutExperience", "screens": [ { "screenType": "DEFAULT", "header": { "logo": "..." }, "content": { "titleOne": "Earn [PotentialRewards]", "contentBlocks": ["Choices", "Terms", "CTA"] }, "footer": { "contentBlocks": [] } }, { "screenType": "REWARDS_ADDED" }, { "screenType": "REWARDS_DECLINED" } ] }
Result Product can configure content; React renders typed blocks; checkout state fills dynamic values like cart total and rewards.
1

Schema owns content

Screens, copy, rich text, terms, choices, CTA blocks, and tiered reward messaging live in Block Builder.

2

Checkout owns data

Cart total, potential rewards, membership price, SKU status, and veil behavior stay close to checkout state.

3

Reducer owns flow

Terms accepted, selected choice, loading, errors, screen transitions, and veil dismissal live in local state.

Block Builder schema

CMS controls the experience structure

Instead of hardcoding one banner, the schema describes the full experience, each screen, the blocks inside each screen, and the styling knobs content editors can adjust.
